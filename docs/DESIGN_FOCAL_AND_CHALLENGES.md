# 设计重点与难点

本文归纳本作品在**设计与实现上的核心着力点**及**已遇到或需持续应对的技术难点**，适用于竞赛申报书「重点、难点及解决方案」章节；与 `docs/DESIGN.md`（总体设计思路）配套阅读。

---

## 一、设计重点

### 1. 以「负责人」为中心的一体化心智

**重点**：产品默认用户是**企业 / 项目负责人**，而非普通「群成员」或「新入职同事」。全员频道、欢迎语、自动回复的提示词与兜底话术均围绕**汇报、执行、尊重称谓**组织，避免智能体把老板误称为新同事，保证叙事与真实使用场景一致。

### 2. 「聊天优先」的组织壳 + 任务闭环

**重点**：底部导航强化 **首页（全员在场）— 工作流 — 布置任务 — 消息 — 我的**，让用户先建立「在组织里说话」的习惯，再自然进入工作台与任务；工作流作为**长期协作容器**，与沟通线程绑定，减少工具跳转带来的上下文丢失。

### 3. 多数字员工的可感知编排

**重点**：全员群播报、用户发言后的多 Agent 回复等场景，需体现**多角色、有节奏、可区分**，而非单模型长文或一次性刷屏。通过**条数上限、时间间隔、轮询刷新列表**等设计，在「像真群」与「性能可接受」之间折中。

### 4. 多端统一与可演示性

**重点**：采用 **uni-app** 统一 App / 小程序 / H5 工程形态，满足比赛多终端演示需求；同时保留**无密钥、无后端弱网**下的本地兜底与静态展示能力，保证答辩现场**可讲、可看、不依赖单一外部服务必达**。

### 5. 业务与智能解耦

**重点**：登录、工作流、消息等业务走**自有后端 REST**；大模型走用户配置的 **OpenAI 兼容接口**，二者在代码层分离（如独立 `clientApi` 与 `openaiCompatible`），便于换模型、换网关与合规说明。

### 6. 可维护的部门与角色矩阵

**重点**：部门详情按 **子组件分支渲染**（`components/department/*`），避免单文件巨型 `v-if` 树，便于扩展新部门、答辩时展示「结构化组织数据」与 UI 的对应关系。

## 二、技术难点与应对思路

### 难点 1：大模型延迟与不稳定对「首屏体验」的冲击

**表现**：全员播报若强依赖单次长请求，易出现长时间无气泡、用户以为卡死。  
**应对**：播报链路**分段异步**；新闻拉取**限时**；模型请求**超时上限**与**本地 / 规则兜底文案**；前端**轮询增量合并消息列表**，先出欢迎与系统提示，再逐条出现 Agent 内容。

### 难点 2：多 Agent 群聊中的幻觉与角色越界

**表现**：模型替他人发言、复述整段他人消息、或在正文中重复写「某某：」与气泡侧名称重复。  
**应对**：**单 Agent 单条 system 约束**；上下文仅取近期若干条；提示词明确禁止代写多人对话；对回复做 **「姓名 + 中英文冒号」前缀剥离**（与 `senderName` / 展示名变体对齐）；播报 JSON 同样约束「正文不以自称开头」。

### 难点 3：负责人身份与称呼一致性

**表现**：通用「同事群」提示易导致「新同事」等错误称谓。  
**应对**：在自动回复与播报生成的 system 中**显式写入**「发言人为老板 / 负责人」及禁止用语；欢迎文案从「欢迎加入」改为「已为您就绪」等**负责人视角**表述。

### 难点 4：国际化插值与模板引擎混用

**表现**：文案中 `{{ count }}` 若与翻译函数内部变量命名冲突，会导致**插值不渲染**，界面上露出字面量。  
**应对**：翻译替换循环使用独立参数名（如 `paramKey` / `paramVal`），避免与翻译键参数同名遮蔽。

### 难点 5：uni-app 与多端差异

**表现**：部分 Vue 能力在小程序端受限（如历史上对动态组件的限制）、自定义导航与原生 Tab 并存、各端安全域与网络策略不同。  
**应对**：按官方文档选用 **自定义导航 + `hideTabBar`** 等组合；H5 用 **Vite 代理** 规避开发期跨域；小程序在开发工具中配置**合法域名**或开发期关闭校验并文档说明。

### 难点 6：本地演示状态与真实接口数据并存

**表现**：未登录或接口失败时，若与已登录逻辑分叉过多，易出现两套 UI 或脏数据。  
**应对**：统一 **请求层错误码处理**（如 401 清会话回登录）；虚拟全员数据与 HQ 会话在 Store 层**集中读写**，减少页面各自维护一份列表。

### 难点 7：外部资讯依赖与合规

**表现**：全员播报若依赖第三方 RSS 聚合服务，存在超时、限流与隐私政策问题。  
**应对**：**并发限时**、失败走无新闻的兜底路径；文档中声明可选服务与赛时建议配置（见 `INSTALL.md`、开源说明文档）。
