# 设计思路

本文从**问题—目标—架构—关键场景—技术取舍**说明本作品的设计逻辑，与 `docs/PRODUCT_PLAN.md`（产品方案与功能清单）互为补充，便于竞赛答辩与后续迭代对齐。**设计重点与实现难点、对策**见同目录 `DESIGN_FOCAL_AND_CHALLENGES.md`。

---

## 1. 设计背景与问题域

### 1.1 用户画像

目标用户包含 **独立接案者、小团队负责人、一人公司老板**：既要对外接单与沟通，又要在内部同时扮演「产品、项目、技术、运维」等多角色，信息分散在私聊、文档与待办工具之间，**认知负荷高、上下文切换频繁**。

### 1.2 核心矛盾

- **协作心智**：真实团队习惯「在群里对齐、在任务里落地」，而传统工具常割裂「IM / 项目管理 / 文档」。  
- **人力边界**：小团队无法 7×24 配齐各职能岗位，但需要**可持续的「组织感」**——既不像纯聊天机器人那样飘，也不像重型 OA 那样重。  
- **智能介入**：大模型能力若仅做成「侧边栏问答」，与业务流程脱节；若完全自动执行，又缺乏**可控性与可追溯性**。

本作品即围绕上述矛盾，做**轻量组织壳 + 可编排数字员工 + 项目流水线**的一体化尝试。

---

## 2. 设计目标与设计原则

### 2.1 目标（可验收）

| 维度 | 目标 |
|------|------|
| **组织感** | 用户打开应用能感知「部门 / 角色 / 全员频道」结构，而非单页工具堆砌。 |
| **任务闭环** | 从「布置任务 → 工作流 → 工作台沟通与进度」形成可演示的闭环。 |
| **人机协同** | 数字员工可参与群聊、播报与建议，关键决策仍由真人（负责人）主导。 |
| **可扩展** | 人设与部门数据可与后端、开源人设库对齐；模型通过 OpenAI 兼容接口替换。 |

### 2.2 原则

1. **聊天优先，流程跟随**：先让用户愿意「说话、看群」，再自然延伸到任务与看板。  
2. **负责人中心**：文案与交互默认 **老板 / 负责人** 视角（称谓、欢迎语、自动回复语境），避免把用户当成「新同事」。  
3. **失败可演示**：网络或模型不可用时，**本地兜底与静态数据**仍能保证页面可浏览、故事可讲清。  
4. **一端多发布**：采用 **uni-app**，兼顾 App / 小程序 / H5 等竞赛常见演示形态，降低环境成本。

## 3. 总体架构思路

### 3.1 逻辑分层

```
┌─────────────────────────────────────────┐
│  表现层（Vue 页面 + 自定义导航/Tab）      │  ← 沉浸聊天、工作台、部门展示
├─────────────────────────────────────────┤
│  领域状态层（virtualTeamStore 等）       │  ← 全员群 HQ、虚拟群、本地会话与播报节奏
├─────────────────────────────────────────┤
│  接入层（clientApi + request + 环境变量） │  ← 鉴权、代理、与后端 REST 对齐
├─────────────────────────────────────────┤
│  智能层（openaiCompatible + 配置）      │  ← 可插拔模型；与业务解耦
└─────────────────────────────────────────┘
```

**要点**：业务接口与 LLM 调用**分离**——登录、工作流、消息走自有后端；生成类能力走用户配置的兼容接口，便于赛方环境与合规说明。

### 3.2 与后端、人设素材的关系

- **业务数据**（用户、Agent 列表、工作流、消息等）以后端为准，前端做展示与编排。  
- **部门与角色展示**可结合本地 `data/` 与接口字段；人设深度可与 **agency-agents 类开源素材库** 在后端对齐（详见 `docs/OPEN_SOURCE_AND_COMPONENTS.md`）。

## 4. 信息架构与导航设计

### 4.1 底部主导航（Tab）

采用 **「首页—工作流/项目—布置任务—消息—我的」** 心智：

- **首页**：全员指挥面 + 组织入口（高频进入、强「在场感」）。  
- **工作流 / 项目**：中长期协作容器列表。  
- **布置任务**：从「想做事」直达下发，降低路径深度。  
- **消息**：跨工作流线程聚合，符合 IM 习惯。  
- **我的**：账号、模型与安全设置。

### 4.2 页面流

- **部门详情**：按部门拆分展示组件（`components/department/*`），避免单页巨型条件渲染，便于维护与答辩时展示「可扩展部门矩阵」。  
- **工作台**：任务 / 沟通 / 图谱 / 动态等多视图收敛在同一项目空间，减少「跳转到另一个产品」的断裂感。  
- **聊天**：自定义导航与主题色与微信系习惯接近，降低学习成本。

## 5. 核心场景设计

### 5.1 全员群（HQ）

**意图**：模拟「全员站会 + 资讯同步」的高频场景，让数字员工**可见、可回复、有节奏**，而不是静默列表。

设计要点：

- 进入时先清空本会话、写入欢迎与系统说明，**避免白屏等待**。  
- 播报与模型拉取**异步 + 轮询增量刷新**，控制气泡间隔，兼顾「有反馈」与「不刷屏」。  
- 用户发言后，**有限数量 Agent 顺序回复**，并约束提示词：尊重负责人身份、**不在正文中重复发送者姓名**（气泡侧已展示）。

### 5.2 工作流与「布置任务」

**意图**：把「接到的活」变成可跟踪的流水线，与聊天线程绑定。

- 工作流作为 **协作容器**；工作台内任务、沟通、时间线共用同一上下文。  
- 「布置任务」独立入口，强化 **从意图到指令** 的动作，便于演示「老板一句话 → 结构化工单」的叙事（具体以后端指令模型为准）。

### 5.3 消息与群设置

- 消息列表支持常见 **会话管理**（如置顶、删除），减轻多项目并行时的噪音。  
- 聊天设置中区分 **虚拟群 / 全员群 / 成员列表**，与后端 `kind`、`id` 等参数对齐，保证路由与状态一致。

---

## 6. 多 Agent 与人机边界

- **角色提示（System Prompt）** 按「单 Agent、单气泡」约束，减少一人多角、代他人发言等幻觉。  
- **群上下文** 以近期消息截断传入，平衡效果与 token。  
- **边界**：敏感或不可逆操作依赖后端权限与人工确认（产品层在 `PRODUCT_PLAN` 中延伸）；前端侧重 **提示与展示**，不越权替用户承诺。

## 7. 技术选型理由（摘要）

| 选型 | 理由 |
|------|------|
| **uni-app + Vue3** | 一套工程多端运行，贴合比赛演示与真实上线路径。 |
| **Vite** | 开发反馈快，与环境变量、代理配置结合清晰。 |
| **OpenAI 兼容协议** | 厂商中立，便于切换云厂商或校内推理网关。 |
| **本地 Store + Storage** | 演示与弱网场景下保证 HQ、虚拟群等体验连续。 |

---

## 8. 体验与工程细节

- **国际化**：`utils/lang.js` 集中管理文案，关键插值避免与代码变量同名污染（如副标题 `count`）。  
- **性能**：播报条间延迟、请求超时、新闻拉取上限等可配，避免「一分钟无响应」的体感。  
- **一致性**：发送者展示名与正文去重前缀策略统一在 `virtualTeamStore` 等模块处理，减少各页复制粘贴逻辑。

---

## 9. 演进方向（与设计一致性）

在不变更「负责人中心 + 聊天优先」的前提下，可优先：

1. 项目群与 **客户可见边界** 的显式建模；  
2. **@Agent / @人** 分流与采纳标记，强化人机决策链展示；  
3. 与后端 **人设 ID、部门 slug** 的全量对齐表，减少前端硬编码。
